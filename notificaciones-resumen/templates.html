Notas de hand-off para ingeniería

Las tarjetas siguientes explican cómo construir las notificaciones desde el JSON del detector (BADS). No son parte del mensaje que ve el usuario. En orden:

  1. 0 · Anatomía / flujo end-to-end — de la respuesta de BADS hasta el mensaje en Slack (las 6 etapas).
  2. 1 · Mapa de parseo del JSON → notificación — la base: qué campo del payload va a cada parte (con ejemplo real y pseudocódigo).
  3. 2 · Slack · contrato de implementación (KPI vs resumen) — cómo se construye y se envía; decisiones v1.
  4. 3 · Imagen / gráfico del incidente — cómo va la gráfica (solo email en v1).
0 · Anatomía: de la respuesta de BADS a la notificación (flujo end-to-end)

La clave es separar 3 cosas: el QUÉ (el bads_payload del detector), el A QUIÉN/DÓNDE (la config: canal + slack_id), y el CÓMO se ve (Block Kit). El detector no sabe de Slack; Slack no sabe del bads_payload. En el medio, un servicio de notificaciones traduce. 

[1] Detector BADS  ──▶  [2] JSON bads_payload  ──▶  [3] Servicio notif.  ──▶  [4] Block Kit  ──▶  [5] Slack
    genera incidente        status: OPEN,              parsea + arma            blocks[]            renderiza +
    emite evento            findings,                  (buildBlocks)            (texto verbatim)    tag <@slack_id>
                            recommended_actions              ▲
                                                  Config del KPI: canal + slack_id
  1. 1 · Se genera el incidente — El detector BADS (workflow_type: ingestion_analysis) crea los findings con IA, arma el bads_payload (schema_version 1.7), persiste el incidente (status: OPEN) y emite un evento (event_type: com.simetrik.bads.incident.escalated / .enriched).
  2. 2 · Sale como JSON — el mismo payload que compartiste. Dos vías: push (el evento dispara la notificación de KPI en tiempo real) y pull (GET results[] paginado, filtrando status: OPEN + día, para el resumen).
  3. 3 · Quién la parsea — un servicio de notificaciones (backend), no el front ni Slack. Lee la config (canal + slack_id), toma el bads_payload, elige idioma, resuelve [ref_id] con references[] y llama a buildBlocks(payload, lang) (misma función para KPI = 1 incidente y resumen = N agrupados).
  4. 4 · A dónde se parsea — a Block Kit JSON (blocks[]: section/context/divider/actions). Texto verbatim; botones desde recommended_actions; mención <{slack_id}>.
  5. 5 · Cómo lo interpreta Slack — Slack solo entiende Block Kit, no el bads_payload; por eso el builder traduce. Renderiza secciones/contexto/botones; <@U0A87…> se convierte en tag real → por eso se usa el ID de Slack, no @nombre (un @nombre por API llega como texto plano).
  6. 6 · Cómo llega al destino — el servicio publica (chat.postMessage / notification center) con channel (de la config), blocks (el Block Kit) y el slack_id para etiquetar (de la config).
Dónde encaja la config de la UI: el canal y el ID para etiquetar no vienen del payload — son el A QUIÉN/DÓNDE. El builder recibe (payload, config) con config = { channel, slack_id, lang, template }: el payload define el contenido, la config define el destino y a quién se arroba.
⚠️ Único punto abierto (con Iván): cómo se envía el Block Kit — template parametrizado registrado en el notification center vs. Block Kit completo inline por evento. No bloquea: el buildBlocks es el mismo; eso es solo el adaptador de envío final.
1 · Parseo del JSON de anomaly → notificación (mapa de campos)

La respuesta es paginada: results[], donde cada item es un incidente. El render filtra → agrupa → mapea, sin reescribir texto. Idioma: elegir la clave es/en/pt según preferencia del usuario (fallback en).

Campo del JSONDónde se usa / regla
results[].statusFiltro: solo OPEN.
…scope.coverage_dates[]Filtro por día del resumen (día anterior).
…scope.resource_refs[]Agrupación. resource_name = encabezado de grupo · resource_id + resource_type (native/oc_chart) = metadata.
…findings[].title[lang]Título de la tarjeta (verbatim).
…findings[].summary[lang]Narrativa de la tarjeta (verbatim). Aquí viven los números (no hay campos de métrica sueltos).
…findings[].severityBadge + color: URGENT → rojo · REQUIRES_ATTENTION → ámbar.
…findings[].confidenceConfianza numérica (0–1).
…findings[].problem_categoryChip de categoría (MISSING_FILE, VOLUME_VARIATION, …).
…findings[].produced_atTimestamp del incidente.
…hypothesis.confidenceEnum: INDICATIVE / PROBABLE / CONFIRMED.
…recommended_actions[]Botones (ordenar por priority). Ver resolución abajo.
references[] (por objeto)Resuelve los marcadores [ref_id] del texto. Cada finding/hypothesis/acción trae su propio array.
affected_dashboards_count / blast_radius[]Chip "afecta N dashboards" + detalle de KPIs/charts afectados.
  1. Resolución de [ref_id]: el texto trae tokens como [1], [2]. Por cada token, buscar en el references[] de ese mismo objeto el item con ese ref_id: si tiene resource_ref → reemplazar por el resource_name (enlazado a su target); si es un action_ref → se omite del texto (va como botón).
  2. Botones desde recommended_actions[]: recorrer references[] y tomar solo los que tengan action_ref. action_ref.label[lang] = texto del botón · action_type = go_to (Abrir) / create_support_ticket (Crear ticket) / send_message (Enviar mensaje) · target = destino · body[lang] = cuerpo prellenado (ticket/mensaje). Las acciones con solo resource_ref no pintan botón.

  && i.bads_payload.scope.coverage_dates.includes(day));

    const p = inc.bads_payload;
    for (const f of p.findings) {
      card.title = resolveRefs(f.title[lang], f.references);   // verbatim + [n]→recurso
      card.body  = resolveRefs(f.summary[lang], f.references); // verbatim
      card.badge = f.severity;             // URGENT | REQUIRES_ATTENTION
      card.conf  = f.confidence;           // 0..1
      card.cat   = f.problem_category;     // MISSING_FILE | VOLUME_VARIATION
      card.hyp   = p.hypothesis?.confidence; // INDICATIVE | PROBABLE | CONFIRMED
    }
    // 3) botones desde recommended_actions (solo refs con action_ref)
    card.buttons = p.recommended_actions
      .sort((a,b) => a.priority - b.priority)
      .flatMap(a => a.references.filter(r => r.action_ref))
      .map(r => ({ label: r.action_ref.label[lang], type: r.action_ref.action_type,
                   target: r.action_ref.target, body: r.action_ref.body?.[lang] }));
    if (inc.affected_dashboards_count > 0) card.blast = inc.affected_dashboards_count;
  }
}

// 4) resolver marcadores [1] [2]… contra el references[] del mismo objeto
function resolveRefs(text, refs = []) {
  return text.replace(/\[(\d+)\]/g, (_, id) => {
    const r = refs.find(x => x.ref_id === id);
    if (r?.resource_ref) return link(r.resource_ref.resource_name, r.resource_ref);
    return ''; // action_ref → fuera del texto, va como botón
  });
}

3 · Imagen / gráfico del incidente (variante con gráfica)

Solo aplica a incidentes con serie temporal (p. ej. VOLUME_VARIATION). El email tiene dos variantes (toggle arriba): con y sin gráfica. En el prototipo el gráfico es un SVG porque corre en navegador; en correo real no se usa SVG ni JS (los clientes los bloquean) — va como imagen PNG renderizada en el servidor.

  1. Render (backend): por incidente se "fotografía" su gráfico. Opción fiel: headless browser (Playwright) del componente real; opción liviana: charts server-side (ECharts SSR, Vega). Renderizar a 2x y mostrar a la mitad del ancho.
  2. Email: embeber con <img> vía adjunto CID (src="cid:chart-<finding_id>") o URL firmada con expiración corta. Nunca bucket público: es data del cliente.
  3. Slack: bloque image de Block Kit apuntando al mismo PNG — pero ver el riesgo de generación de imagen en la nota anterior.
  4. Accesibilidad: imagen estática, siempre con alt descriptivo; la metadata de texto debe entenderse aunque la imagen no cargue (por eso la variante sin gráfica).
